# 使い手目線の設計・テスト心得

Status: ACTIVE GUIDELINE(このリポジトリで実装・レビューをする全エージェントの必読事項。IDEAS ONLY のメモ群とは違い、これは守るべき規約)。CLAUDE.md の運用契約を補完する、UX とテストに特化した心得。マスターの標準要望(何を満たすべきか)は `docs/master-preferences.md` — 本書はその実現手順にあたる。

## 0. この文書が生まれた理由(実例)

過去に別のコーディングエージェントへ任せた結果、ユーザー目線であり得ない実装が実際に起きた:

1. **再生成ボタンが「同じユーザーメッセージの連続送信」として実装された** — 再生成とは「直前のアシスタント回答を差し替える」こと。会話ログにユーザー発言が二重に積まれるのは仕様理解の失敗。
2. **コピーボタンが存在しなかった** — チャットの回答は他所で使うために存在する。コピー導線がないチャットUIは日常使用に耐えない。
3. **「GPTの基本機能がほしい」と言われたのに、そもそも再生成が実装されなかった** — 曖昧な指示を「最小解釈」した結果。

教訓: **曖昧な指示は「小さく作っていい」という意味ではない。基準製品が名指しされたら、その製品の当たり前がそのまま仕様である。**

## 1. 基準製品パリティの原則

ユーザーが「GPTみたいな」「◯◯みたいな」と言ったら、その製品を日常利用している人が**無意識に期待する機能の一覧**を書き出し、それを仕様の出発点にする。実装前に「含める / 後回し(理由付き) / 含めない(理由付き)」を明示して報告する。黙って削るのが最悪(CLAUDE.md「Broad-scope and settings coverage」と同じ規律)。

## 2. 設計の心得

- **実装前に、ユーザーの動作を1文で書く。** 「ユーザーが◯◯すると、△△が見える」。この文が書けない機能は仕様が固まっていない。受け入れ条件はコード構造ではなく**ユーザーに見える挙動**で書く。
- **すべての操作ボタンについて、5状態を設計してから作る:**
  1. 通常クリック
  2. **連打**(2度押し・ダブルクリック)
  3. **処理中**に押した場合(無効化?キュー?無視?)
  4. **失敗後**(データはどうなる?入力は消える?リトライできる?)
  5. **リロード後**(状態は残る?壊れる?)
  再生成の重複送信バグは、状態2・3の設計を飛ばした結果。
- **破壊的操作には確認、失敗時にはデータ保全。** ユーザーの書いた文章を消す実装は、いかなる失敗経路でも許されない(送信失敗で入力文が消える等)。
- **「なんでやねんテスト」= 自分が毎日使う想定で3分触る。** イラッとした点は全部課題。作った本人がイラッとするものをユーザーに出さない。
- 日本語ユーザーの前提を忘れない: IME、日本語ファイル名、長文の折り返し、全角入力。
- 目に優しい前提を忘れない: ダークモードで新機能を必ず確認。まぶしい白フラッシュを出さない。

## 3. テストの心得

- **テスト名はユーザーの文で書く。** `test("regenerate does not duplicate the user message")` — テスト一覧がそのまま仕様書になる。関数名の羅列(`test("clampInt works")`)より、壊れたとき何が困るかが分かる名前を優先。
- **ロジックは純関数に切り出してテストする。** ペイロード構築(`server/openai.mjs`)が現にそうなっている。今後のナッジ判定(`shouldNudge`)、プライベート指標の丸め・派生計算(weight メモ §7)も同じ型に。DOM に絡めて書くとテスト不能になる。
- **ユーザーシナリオのスモーク台本を維持する**(下記 §4)。ユニットテストは「関数が正しい」ことしか言わない。「アプリが使える」ことは通し操作でしか確認できない。
- **バグは二度と戻さない。** 「なんでやねん」を見つけたら、(a) §1 の表か §4 の台本に項目追加、(b) 可能なら回帰テスト追加、の両方をやってから直す。
- 境界での検証を優先: 入力ゼロ文字、超長文、履歴100件、APIキー無し、サーバー停止中、localStorage 破損(壊れたJSON)。

## 4. 5分スモーク台本(リリース前・大きな変更後に手動実行)

1. 起動 → Chat タブで「こんにちは」送信 → 入力欄が空になり、返答(モック可)が出る
2. 送信ボタンを**素早く2連打** → メッセージが1つしか送られない
3. 処理中に再生成・クリアボタンが**無効化**されている
4. 再生成 → ユーザー発言は増えず、回答だけ差し替わる
5. 再生成を**連打** → 1回分しか動かない
6. サーバーを止めて送信 → エラー表示+入力文が消えていない → サーバー再開してリトライ → 成功
7. **日本語IMEで変換中にEnter** → 送信されない。確定後Enter → 送信される
8. リロード → 会話・設定・Lifeデータが残っている
9. ダークモード切替 → 全タブ表示確認(まぶしい要素・読めない文字がない)
10. 設定でモデルを reasoning 系に変更 → temperature/top-p が無効化され、説明が出る
11. Diary: 「書いてもらう」→ エントリが表示される(モック可)。ボタン連打で1回しか動かない。削除は確認つき
12. News: タブを開いただけでは取得・AI分類が始まらない → Refresh → カードが出る(またはフィードエラーが正直に表示される)。「隠す」で消え、リロード後も消えたまま。興味トピックの追加・削除ができる
13. 利用料金: APIキーなしのモック送信では件数が増えない。実usageまたはfixtureでは回答に今回額、Settingsに直近・選択期間・全期間が表示され、リロード後も残る
14. 利用料金の日付: 開始日・終了日の両端が集計に入り、開始日を終了日より後にすると明示エラーになる。幅375px・ダークモードでも横にはみ出さない
15. SettingsのBackup: Exportを連打しても1ファイル分だけ動き、`mint-room-backup-YYYY-MM-DD.json`のダウンロード開始が表示される。会話が平文で入る注意と、サーバー側`OPENAI_API_KEY`は含まれない説明が見える
16. Importへ正常ファイルを選ぶ → 本文を出さずカテゴリ件数・新規/上書き/削除/保護件数だけpreview → Cancelでデータが変わらずImportボタンへfocusが戻る。空・壊れたJSON・上限超過はwrite前に拒否される
17. 検証用データをExport → 内容を変更または保存領域を空にする → Import → 復元前バックアップのダウンロード表示を確認して2回目のOK → 即reload後に会話・設定・Life・日記・ニュース・利用料金が復元される。処理中のChat/日記/ニュースがある間は確定できない
18. 破損fixture: 1カテゴリだけ壊れた場合はそのカテゴリを明示スキップして他を救出できる。利用料金は有効な一意イベントの救出件数/除外件数、またはImport時点から0件再開を表示する。未知の将来キー衝突は現在値を上書きしない。幅375px・ダークモードでpreviewが横にはみ出さない

所要5分。**「全部✅」以外の状態でタスク完了を報告しない。** 実行できなかった項目は「未確認」と明記する(CLAUDE.md の検証規律と同じ)。

## 5. 曖昧な指示を受けたときのプロトコル

1. 基準製品・比較対象を特定する(名指しがあればそれ、なければ用途から推定して明記)。
2. 暗黙の期待リストを書き出す(§1 の表のように)。
3. 「今回含める / 後回し / 含めない」を**実装前に**宣言する。
4. 迷ったら「日常的にそれを使う人が初日に気づいて困るか?」で判定。困るなら含める。
5. 指示が曖昧だったことをユーザーの落ち度にしない。**曖昧さを埋めるのが実装者の仕事。** 埋め方を報告すれば、ユーザーは差分だけ直せる。
