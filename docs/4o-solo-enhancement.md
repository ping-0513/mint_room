# 4o単体を素より賢くする — アイデアメモ

Status: IDEAS ONLY(実装約束ではないメモ)。強い推論モデル(GPT-5.5T的なもの)を併用する設計は `docs/4o-thinking-orchestration.md` にある。**このメモは「4o的な会話モデル1つだけ」で、素の状態より賢く感じさせる工夫**を集めたもの。両者は排他ではなく、ここの工夫は将来の二段構えでもそのまま効く。

実装時の共通ルール: プロンプト・ペイロード構築はすべて `server/openai.mjs` に集約(UIコンポーネントで組み立てない)。会話の温かさを壊さないこと。優先度: ★=効果大かつ軽い、☆=効果大だが重い。

---

## A. 呼び出し1回のまま賢くする(追加コストほぼゼロ)

賢さの体感は「モデルの地力」より「モデルに何を見せるか」で決まる部分が大きい。ここが一番費用対効果が高い。

### A-1. ★ 現在日時・タイムゾーンの注入

4oは現在時刻を知らない。「今日って何日?」「あと何時間で寝る時間?」に答えられない素の状態は、それだけで「賢くない」印象になる。クライアントのローカル時刻+タイムゾーンをリクエストに含め、サーバー側で instructions に1行足すだけ(例: `Current local time: 2026-07-20 (Sun) 23:10 JST`)。就寝ナッジ設計の前提でもある。

### A-2. ★ 生活データの文脈注入(グラウンディング)

Lifeタブのタスク・買い物・服薬状況、カレンダーの直近予定をコンパクトに instructions へ載せる(例: 「未完了タスク3件: …。服薬: 朝○済/夜□未」)。「私のことを分かってくれている」感は賢さの体感そのもの。注意点:

- トークン節約のため件数上限つき要約形式で(全件ダンプしない)。
- プライバシー設定(送信オフ)を必ず用意 — 送る内容が増えるほどオプトアウトが重要。

### A-3. ★ 開発者指示テンプレの質を上げる

素の4o的モデルは「浅く早く同意する」傾向がある。既定の developer instructions(ユーザーが空欄のときのアプリ既定値)に、行動品質の指示を入れる:

```text
- 曖昧な依頼は推測で走らず、1つだけ確認質問をする。
- 事実に自信がないときは、推測であることを自然な言い方で添える。
- 手順・計画・比較を求められたら、短い見出しか箇条書きで構造化する。
- 数値・日付の計算は途中経過を短く書いてから結論を言う。
- 長文が必要な時以外は簡潔に。ただし冷たくしない。
```

「途中経過を書いてから結論」は非推論モデルの正答率を実際に押し上げる古典的手法(step-by-step)。出力に薄く構造が出るが、Markdownレンダリング(feature-ideas §1)と組み合わせれば読み味はむしろ良くなる。

### A-4. ★ タスク種別ごとのパラメータ自動プリセット

雑談は temperature 高め、計画・計算・事実確認は低め、が定石。メッセージをアプリ側の軽いヒューリスティック(モデル呼び出しなし: キーワード・長さ・疑問符など)で分類し、送信パラメータをプリセット切替する。ユーザーが手動設定した値は常に優先。設定UIには「自動調整(おすすめ)」トグルとして出す。

### A-5. 会話の長期一貫性(要約引き継ぎ)

`historyLimit` を超えた古い履歴を切り捨てる現状は「さっき言ったのに忘れる」= 賢くない体感に直結する。切り捨て分を要約して instructions に引き継ぐ。要約生成も4o自身にやらせる(B節の低頻度呼び出し)か、まずは「直近の要点をユーザーが固定メモできる欄」でも代替可。

### A-6. やさしい長期メモリの注入

feature-ideas §2 のメモリ(名前の呼び方、好み、苦手なこと)を instructions に載せる。記憶内容の一覧・削除UIとセットで。

### A-7. 計算はアプリがやる

日付差・合計金額・単位換算など、LLMに暗算させると間違える系はアプリ側で計算して「結果を文脈として渡す」。tool use 実装前でも、時刻注入(A-1)と同様の文脈注入で一部先取りできる。

## B. 同じ4oを複数回使う(単体のまま二段構え)

**orchestration 設計の reasoningModel に primary と同じ 4o を設定すれば、追加モデルなしで two-pass が動く**。これがこのメモの核。設計済みの orchestrator(fast_chat / assisted_chat / thinking_response / regenerate_review)はモデルIDが同じでもそのまま成立する。

### B-1. ☆ think-then-answer(隠しメモ → 本回答)

複雑な質問のときだけ: 1回目の4o呼び出しで「計画・論点・落とし穴」を低温度・小トークンの構造化メモとして出させ、2回目で本回答。同じモデルでも「考えてから話す」だけで質が上がる。トリガー条件・スキーマ・失敗時フォールバックは orchestration doc §4/§10 をそのまま使う。

### B-2. draft → 自己批評 → 修正(self-refine)

回答案を作らせ、「この回答の弱点を3つ挙げて改善版を出して」を同じ4oに1回だけ回す。効果はあるがレイテンシ×2なので、「じっくりモード」やregenerate時限定が現実的。批評ループは1周まで(2周目以降は劣化しがち+コスト増)。

### B-3. best-of-N + 自己選択

温度高めで2〜3案生成し、4o自身に最良案を選ばせる。創作系(文章を書いて等)にだけ効く。既定オフの実験的設定でよい。

### B-4. ★ regenerate_review を4o自身で

「再生成」時に、直前の回答の何が悪かったかを4o自身に診断させてから再回答(orchestration §4 の regenerate_review そのまま)。強いモデルがなくても「同じ失敗を繰り返さない再生成」になり、体感がかなり違う。二段構え実装の最初の一歩としてもコスト管理が楽(ユーザーが明示的に押した時だけ発火)。

### B-5. コスト・レイテンシ管理(B節共通)

- 全メッセージには絶対に使わない。トリガー制(長文・計画依頼・「ちゃんと考えて」・再生成)+ セッション内回数上限。
- 隠しメモは `max_output_tokens` 小さめ(300〜800、orchestration doc と同じ目安)・低温度・構造化出力。
- 失敗したら黙って通常回答にフォールバック(エラーを見せない)。

## C. モデルを呼ばずに賢く見せる(アプリロジックのみ)

- **JSON/形式の検証と自動リトライ1回** — JSONモード時に壊れた出力をアプリが検知し、エラー内容を添えて1回だけ再依頼。「壊れた出力がそのまま出る」体験を消す。
- **時刻ベースの純ロジックナッジ** — 就寝・服薬ナッジはモデルなしで動く設計済み(orchestration §5)。「気づいてくれた」体感はモデルの賢さと区別がつかない。
- **送信前の軽い入力補助** — 空メッセージ・重複送信ガード(実装済み)に加え、「さっきと同じ質問です」検知で再生成を提案する等。

## 効果の目安と限界(正直に)

期待できること: 文脈把握・一貫性・構造化・計算まわりの体感改善は大きい。特に A-1/A-2/A-3 は「同じモデルなのに別物」レベルの体感差が出やすい。

期待できないこと: 4o単体は本物の推論モデルにはならない。多段呼び出し(B節)でも、数学的に難しい問題・長い厳密な推論では専用推論モデルに劣る。だからこそ orchestration 設計では reasoningModel を差し替え可能にしてあり、**まず B 節を「4o×2」で実装 → 後日リーズニングモデルが使えるようになったら設定変更だけで強化**、が推奨経路。

レイテンシの現実: B節はどうしても応答が2倍前後遅くなる。「速い4oらしさ」が売りのアプリなので、既定は A+C のみ、B はトリガー時だけ、を守ること。

## 実装順のおすすめ

1. A-1 時刻注入 + A-3 既定指示テンプレ(どちらも `server/openai.mjs` の instructions 合成に足すだけ)
2. A-2 生活データ注入(プライバシートグル同時実装が条件)
3. B-4 regenerate_review(4o×2の最初の一歩、ユーザー操作起点でコスト安全)
4. A-4 パラメータ自動プリセット
5. A-5 要約引き継ぎ + B-1 think-then-answer(orchestrator Stage 3〜4 と合流)

1〜4 は DB・認証・新規依存なしで実装可能。着手時は `docs/agent-handoff.md` に実装済み/未実装を反映すること。
