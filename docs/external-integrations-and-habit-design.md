# 外部連携と習慣デザイン — アイデアメモ

Status: IDEAS ONLY(実装約束ではないメモ)。関連: `docs/feature-ideas.md`(全般バックログ)、`docs/4o-solo-enhancement.md`(文脈注入)、`docs/4o-thinking-orchestration.md`(ナッジ設計)。

このメモは2つのテーマを扱う:

1. **記憶・記録・予定を内部で抱え込まず、Obsidian や Google カレンダー等の外部に置くことで、安定性と正確性を上げる設計**
2. **人間側が面倒になって使わなくなる事故を防ぐ、報酬感と継続性の構造**(時間を守る・食事を管理する行動が続く仕組み)

優先度: ★=効果大かつ軽い、☆=効果大だが重い。

---

# 第1部: 外部連携 — 「内部で全部やらない」設計

## 1-0. 基本原則: LLMは記憶しない、参照する

- **信頼できる事実(予定・記録・記憶)は外部の正規のストア(source of truth)に置き、LLMには毎回それを見せる。** LLMの中に事実を溜め込まない。
- 得られるもの:
  - **正確性** — 繰り返し予定・タイムゾーン・日付計算は Google カレンダーが正しくやる。LLMに暗算・記憶させない(幻覚の根本対策)。
  - **安定性** — localStorage 全依存の脆さ(ブラウザデータ消去で全消失)から脱出。記録はユーザーの手元のファイルに残る。
  - **信頼** — AIの「記憶」が人間可読な Markdown なら、ユーザーが監査・修正・削除できる。ブラックボックスの不気味さが消える。
  - **開発コスト減** — 自前DB・同期システムを作らずに済む(CLAUDE.md の「DBなし」方針とも一致)。
- 責務の分担表(どの層が何を所有するか):

| 情報 | 正となる場所 | LLMの役割 |
|---|---|---|
| 予定・繰り返しイベント | Google カレンダー(外部) | 読んで言葉にする・追加を提案する |
| AIの長期記憶・日々の記録 | Obsidian vault(ローカルMD) | 読んで文脈にする・追記案を出す |
| 今日のタスク・服薬・買い物 | アプリ(localStorage→将来MD) | 読んで気づく・チェックを促す |
| 時刻・日付計算 | アプリの純ロジック | 計算しない(結果を受け取るだけ) |
| 会話 | アプリ | 主戦場 |

## 1-1. Obsidian 連携(AIの記憶・ユーザーの記録)

**好条件:** 本アプリのサーバー(`server.mjs`)はユーザーのマシンで動くローカル Node なので、**Obsidian vault(ただの Markdown フォルダ)をサーバー側から直接読み書きできる**。クラウド不要・OAuth不要・プライバシー最良。

### 連携の階段(下から順に実装)

1. ★ **手動エクスポート/インポート** — 会話要約・記録を Markdown でダウンロード。ユーザーが vault に置く。実装最小、まず「データはあなたのもの」を体現。
2. ★ **vault パス直結(読み)** — 設定でフォルダパスを指定(例: env `OBSIDIAN_VAULT_PATH` または設定UI)。サーバーが指定ノート(例: `mint-room/memory.md`, `mint-room/profile.md`)を読み、会話の文脈に注入(4o-solo A-2/A-6 と合流)。**読み取り専用から始める**のが安全。
3. ☆ **vault 直結(書き)** — デイリーノート追記(`mint-room/daily/2026-07-20.md`)と記憶ノート更新。書き込みは**追記のみ(append-only)を原則**とし、既存内容の書き換え・削除はしない。上書きが必要な記憶更新は「差分プレビュー → ユーザー承認」を挟む。
4. **Obsidian Local REST API プラグイン経由**(コミュニティプラグイン)や **Obsidian URI**(`obsidian://new?...`)— vault 直結で足りるため優先度低。ユーザーが既にプラグインを使っている場合の代替経路として記録のみ。

### 記憶ノートの設計

- **人間可読が絶対条件。** AIの記憶 = ユーザーが普通に読める Markdown。frontmatter(YAML)に機械可読の最小メタだけ置く。
- 例の構成:
  - `profile.md` — 呼ばれたい名前、好み、苦手、体質メモ(ユーザー編集が主、AIは提案のみ)
  - `memory.md` — AIが会話から学んだこと(1行1事実、日付つき、AIが追記案 → 承認で確定)
  - `daily/YYYY-MM-DD.md` — その日の会話要約・食事・睡眠・できたこと(追記のみ)
- **削除権はユーザーに。** ノートから行を消せば AI は次回からそれを知らない — この単純さが信頼になる。
- 安全境界: vault パス外へのアクセス禁止(パス正規化必須)、`.env` 等の除外、書き込みは `mint-room/` サブフォルダ限定を既定に。

## 1-2. Google カレンダー連携(予定管理)

OAuth は重い。**OAuth なしで読み書きの大半を実現する2段構え**を先に:

1. ★ **読み: 非公開 iCal URL(ICS)** — Google カレンダーの「限定公開URL(iCal形式)」を設定に貼ってもらい、サーバーが定期取得してパース。OAuth 不要で本物の予定が読める。繰り返し予定の展開・タイムゾーンは ICS データに従う(RRULE 展開は既存実装の慎重な移植か、小さな依存の追加承認が必要 — 要相談ポイント)。URL は秘密情報扱い(サーバー側保管、ログに出さない)。
2. ★ **書き: テンプレートURL方式** — `https://calendar.google.com/calendar/render?action=TEMPLATE&text=...&dates=...` 形式のリンクをアプリが生成し、**ユーザーがクリックして自分で追加**。OAuth不要・誤登録なし(人間が最終確認)・実装は文字列組み立てのみ。
3. ☆ **将来: OAuth 両方向同期** — 自動登録・更新・削除が必要になったら。クライアントID管理・同意画面・トークン保管という別次元の重さなので、1・2で不満が出てから。Stage 7(承認後)扱い。

効果: カレンダータブが「本物の予定」を表示でき、ナッジ(routine_watch)が実データで動く。「明日9時に病院あるよ、今日は早めに寝る?」が幻覚なしで言える。

## 1-3. その他の外部化候補(メモのみ)

- タスク: Todoist 等はAPIキー1本で読めるものもあるが、まずは Obsidian のタスク記法(`- [ ]`)で兼ねる方がストア数が増えない。
- 食事・健康記録: 専用アプリ連携より「デイリーノートに1行追記」(第2部参照)が本アプリの性格に合う。
- 原則: **外部ストアは2つまで(Obsidian + Google カレンダー)から始める。** 増やすほど設定・故障点・プライバシー説明が増える。

# 第2部: 報酬感と継続性 — 使われ続ける構造

## 2-0. 基本原則: 摩擦は5秒以内、罪悪感はゼロ

続かない原因は「記録が面倒」「サボると責められる(気がして開かなくなる)」の2つがほぼ全て。だから:

- **1操作5秒以内** — 記録はワンタップかチャット1行。フォームを作らない。
- **失敗を数えない** — 「できなかった日」をUIが強調しない。できたことだけ光る。
- **休んだ後こそ優しく** — 数日ぶりに開いたら「おかえり🌿」だけ。「7日間記録がありません」は絶対に出さない(離脱の最大要因)。
- **全部オプション** — 記録も習慣機能も使わなくてもチャットアプリとして完結する。

## 2-1. 報酬感の設計(mint room の世界観で)

- ★ **チェック時のマイクロ報酬** — タスク・服薬・就寝チェックの瞬間に、小さなきらめきアニメーション+ミントカラーの満足感ある演出(0.5秒以内、reduced-motion 設定を尊重)。「押すと気持ちいい」が最初の報酬。
- ☆ **部屋が育つ(アンビエント報酬)** — ポイントやランクではなく、**続けていると mint room 自体が少しずつ心地よくなる**方向。例: 窓辺の植物が育つ、小物が増える、照明の雰囲気が変わる。数値化しない・失っても枯れない(成長が止まるだけ)。競争や焦りを生まない報酬。
- ★ **具体的な承認の言葉** — AIの褒め方は汎用語(えらい!)より実データ参照(「3日続けて23時台に寝られてるね。目の調子どう?」)。外部記録(第1部)があるから言える台詞 = 連携の価値が報酬として還元される。
- ★ **ソフト・ストリーク** — 連続日数チェーンは切れた瞬間に離脱を生む。「今週 5/7 日」の週率表示にする。0にリセットされる体験を構造的に排除。
- ☆ **週次ふりかえりカード** — 週1回、記録から優しい要約(「朝が先週より安定」「水曜は疲れてたみたい」)。**記録する意味が見える = 記録行動自体の報酬**。Obsidian の週次ノートにも保存でき、資産になる。

## 2-2. 時間を守れる構造(命令ではなく契約)

AIに「寝ろ」と言われて寝る人はいない。効くのは**自分で決めた計画を、いいタイミングで思い出させてもらう**構造:

- ★ **if-then 事前約束(実行意図)** — 設定時にユーザー自身が「23時になったら→お風呂」「朝ごはんの後→薬」を決める。ナッジは常に「あなたが決めた◯◯の時間だよ」という形式。命令ではなく代理リマインドになるので反発が起きにくい。
- ★ **会話の「区切り支援」** — 就寝時刻が近いとき、AIが会話を**きれいに畳む**手伝いをする: 「いいところで区切ろうか。ここまでの話、メモに残しておくね」→ 要約をデイリーノートへ保存 → 「続きは明日聞かせて」。**中断ではなく完了の感覚**にする(やりかけ感が残ると人は寝ない)。
- ★ **二択ナッジ** — 「今からおやすみモードにする?それとも15分後?」。自己決定感が残る二択は、一方的リマインドより従いやすい。「無視」も常に選べる(ボタンを押さなくても責めない)。
- **おやすみモード = ごほうび化** — 突入すると: テーマが目に優しく減光 → 今日できたことが3つ表示 → 明日の予定と持ち物が1枚に → 「おやすみ🌙」。寝ることが「切り上げ」ではなく「今日の完成」になる演出。
- **朝はハードルゼロ** — 起床時間帯に開くと挨拶+今日の1枚(予定・薬・タスク上位3件)。まず**見るだけでいい**。行動要求は朝イチにしない。

## 2-3. 食事管理の構造(ケアであって監視ではない)

- ★ **チャットが記録装置** — 「ラーメン食べた」と話すだけで、AIがデイリーノートに `12:30 昼: ラーメン` を追記(tool use、追記は確認後 or 追記のみ自動+取り消しボタン)。**専用フォームを開かせない**ことが継続の全て。
- ★ **まず「食べたかどうか」だけ** — 第一目標は欠食の検知(「今日お昼まだみたい。何か食べられそう?」)。カロリー・栄養素の管理は任意の後段。規則性 > 内容、の順で価値を出す。
- **写真ワンタップ記録** — 画像入力(実装済みロードマップ)と合流: 食事の写真を送る→AIが1行記録。内容推定は「たぶん」を付けて控えめに。
- **安全境界(重要):**
  - カロリーの叱責・体型への言及・過度なダイエット励行はしない。**恥や罪悪感を使う食事フィードバックは禁止**(摂食問題を悪化させるリスクがあり、本アプリの「優しい部屋」という核にも反する)。
  - 医療・栄養アドバイスではなくセルフケア記録である旨を明示。既存の服薬機能と同じ整理(チェックリストであって医療ではない)。
  - 食事記録はデフォルトでローカル/vault のみ。背景モデルへ送るかは既存のプライバシートグルに従う。

## 2-4. 継続ループ(全体構造)

```
感知(時刻・カレンダー・記録) → 提案(優しい二択ナッジ)
   ↑                                    ↓
ふりかえり(週次カード)  ←  記録(外部MD・追記のみ)  ←  行動(1タップ/チャット1行) → マイクロ報酬
```

- 感知と記録は決定論的なアプリロジック+外部ストア(第1部)。**LLMは「提案の言葉選び」と「ふりかえりの文章」だけを担当** — ここでも内部で全部やらない原則が安定性を支える。
- ループのどこで離脱しても壊れない: 記録しなくてもチャットは楽しい、ナッジを全部無視しても翌週また静かに1回だけ提案される。

## 実装順のおすすめ(第1部・第2部合わせて)

1. ★ Markdown手動エクスポート(1-1の階段1)+ チェック時マイクロ報酬(2-1)— どちらも軽く、価値がすぐ見える
2. ★ Obsidian vault 読み取り連携 → 文脈注入(1-1の階段2、4o-solo A-2と同時実装)
3. ★ ICS読み取りでカレンダー実データ化(1-2)+ 二択ナッジ(2-2、orchestration Stage 2と合流)
4. ★ カレンダー追加テンプレートURL(1-2)+ ソフト・ストリーク表示(2-1)
5. ☆ vault 追記(デイリーノート)+ チャットからの食事1行記録(1-1階段3 + 2-3)
6. ☆ おやすみモードのごほうび化 + 週次ふりかえりカード
7. 将来(承認後): Google OAuth 両方向同期、部屋が育つ演出の本格版

注意: 2 以降はローカルファイルアクセスという新しい安全境界(パス制限・追記のみ・承認フロー)を伴うため、着手前に `docs/agent-handoff.md` に境界設計を書いてレビューを通すこと。ICS パースで依存追加が要る場合は承認を得ること(CLAUDE.md の依存追加ルール)。
