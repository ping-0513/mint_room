# Claude × Codex 協働プロトコル

Status: ACTIVE GUIDELINE(両エージェント必読)。前提: マスターの環境では Codex がほぼ使い放題なので、**Codex の呼び出し回数・思考量は節約しない**。常に最大の思考努力(reasoning effort 最大)で走らせてよい。節約するのはマスターの確認の手間と、リポジトリの秩序だけ。

## 0. 役割分担(公開情報で裏付けられた得意分野に基づく)

| 領域 | 主担当 | 理由 |
|---|---|---|
| 仕様の解釈・曖昧さの解消・スコープ決定 | **Claude** | 複数の妥当な選択肢がある判断・トレードオフの言語化が得意。Sol系は自信満々に難所を素通りする弱点が文書化済み |
| アーキテクチャ・設計文書・ガイドライン管理 | **Claude** | ACTIVE ガイドライン群の整合性維持は判断業務 |
| UI/UX の見た目の決定 | **Claude** | マスターの好みは Claude 系の出力(`docs/ui-recipes.md` に言語化済み。Codex も従えば再現可能) |
| チケット化された実装 | **Codex(思考最大)** | 実行力は強い。スコープが明確なら最高品質で書ける |
| テストの量産・エッジケース網羅 | **Codex(思考最大)** | 思考コストをかけるほど網羅が効く領域。使い放題が最も活きる |
| ログ・大量出力の解析、機械的な一括変更 | **Codex** | 量の仕事 |
| 相互レビュー | **両方** | モデル系統が違う=盲点が違う。重要変更は相手モデルの目を必ず通す |
| 最終責任・マージ判断 | **Claude**(マスターの意向が常に最上位) | 誰かが決定に責任を持つ必要がある |

## 1. 通信路はリポジトリ(`docs/tasks/`)

ClaudeとCodexは別セッション・別環境で動くため、**会話ではなくファイルで協働する**。

- Claude がチケット(`docs/tasks/NNN-slug.md`)を書く → マスターが Codex にチケットを渡す(「docs/tasks/001 をやって」だけでよい)→ Codex が実装し、同じファイルの報告欄を埋めてコミット → Claude(または相手モデル)がレビュー欄を埋める → DONE。
- チケットの書式・状態遷移は `docs/tasks/README.md` に固定。**書式を崩さない**(次のエージェントの読解コストが上がる)。
- 会話でしか伝わっていない決定は存在しないのと同じ。決定は必ずチケットか handoff に書く。

## 2. Codex 使い放題を活かす運用パターン

節約不要という前提で、普通ならコスト的に贅沢な手を標準にする:

1. **思考は常に最大** — チケットに明記済み。速さより賢さを取る。ただし思考最大でも AGENTS.md Rule 5(2回失敗で停止)は有効 — 思考量と粘着は別物。
2. **実装前メモ** — 中規模以上のチケットでは、Codex はコードを書く前にチケットの「実装前メモ」欄に計画・懸念・代替案を書いてから着手する(思考を文書として回収する)。
3. **2案並走** — リスクの高い実装は Claude がチケットに `variant: A/B` を指定し、Codex が2通り実装(別コミット)→ Claude が比較して選ぶ。捨てた案もチケットに記録。
4. **テスト先行の別働隊** — 機能実装と同時に、別チケットで「その機能を壊すテストを書け」を走らせられる(書く人と壊す人を分ける)。
5. **健康診断スイープ** — 手が空いたら「`npm test`・`npm run check`・スモーク台本・パリティ表の鮮度確認」だけのチケットを回してよい。

## 3. 境界(衝突を防ぐ約束)

- **Codex は ACTIVE ガイドライン(`AGENTS.md`, `CLAUDE.md`, `docs/master-preferences.md`, `docs/ux-design-testing-principles.md`, `docs/ui-recipes.md`, 本書)の本文を直接編集しない。** 変更提案はチケットの報告欄に書き、Claude が反映する(Sol系の意図超過への構造的対策)。
- **例外 = 生きた記録簿:** `ux-design-testing-principles.md` 内のパリティ表(§1)とスモーク台本(§4)、および `docs/agent-handoff.md` は実装状態の記録なので、**自分の変更が影響するときは実装者(Codexも)が更新する義務がある**。ガイドラインの「ルール本文」と「記録簿」を区別すること。
- `docs/agent-handoff.md` は削除はせず追記・修正のみ。
- 同時作業の衝突回避: 1チケット=1担当。チケットの Status を IN PROGRESS にしてから作業する。
- 意見が割れたら: 事実(テスト・計測・公式ドキュメント)で決まるものは事実で。趣味・判断のものは Claude、最終的にはマスター。

## 4. 相互レビューの観点(レビュー欄に書くこと)

- スコープ: チケット記載のファイル以外を触っていないか(AGENTS.md Rule 1)
- 検証: `npm test` / `npm run check` / 該当スモーク手順の実行結果が報告されているか。**未実行なのに DONE になっていたら差し戻し**(Rule 2)
- UI: `docs/ui-recipes.md` 準拠(新しい色・角丸・アニメを発明していないか、ダークモード確認したか)
- コメント: 新規コードのコメントが日本語で意図を説明しているか
- 記録: handoff・パリティ表が更新されているか

レビューで直させるより、チケットの受け入れ条件を最初から明確に書く方が安い。差し戻しが2回続いたらチケットの書き方を疑うこと(Claude 側の責任)。

## 5. マスターの操作コスト最小化

マスターがやることは原則2つだけ:

1. Codex に「docs/tasks/NNN をやって」と渡す
2. DONE になったものを眺めて、気に入らなければ一言書く

割り込み質問はチケットの「質問」欄に集約され、マスターは気が向いた時にまとめて答えればよい。緊急でない質問で作業を止めない(質問を書いたら、影響しない範囲で作業は続ける)。
