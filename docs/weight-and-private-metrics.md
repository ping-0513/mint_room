# 体重管理と「AIに見せない」設計 — プライベート指標メモ

Status: IDEAS ONLY(実装約束ではない設計メモ)。関連: `docs/external-integrations-and-habit-design.md`(記録・報酬)、`docs/4o-solo-enhancement.md`(文脈注入 A-2)、`docs/feature-ideas.md`。

要件: **体重管理をしたい。ただし生の体重をAI(4o)に見られたくない。** モデルには「ダイエット開始日から何割減ったか」だけが分かればいい。

これを1機能として作るのではなく、**「生データはローカルだけ、AIには派生値だけ」という汎用プライバシー変換レイヤー**として設計する。体重は最初のユースケース。

---

## 1. 基本原則: カーテン越しコーチ

- **生値(kg)はデバイスから出ない。** localStorage(将来は Obsidian vault)に保存。APIリクエストのどの層にも含めない — サーバーで隠すのではなく、**そもそも送信ペイロードに載せない**(変換はクライアント側で完結)。
- **AIに渡るのは相対値・傾向・進捗率だけ。** AIは「開始から2.5%減、ゆるやかに減少中」は知っているが、何kgかは永遠に知らない。トレーナーがカーテン越しに応援しているイメージ。
- **ユーザー自身は全データを見られる。** 自分のグラフには実数kg表示。AIと画面表示は別物。
- **「AIに見えているもの」を常に確認できる。** 設定に「AIビュー」プレビューを置き、共有される文字列をそのまま表示(下記の例)。隠す機能こそ、何を隠せているかの透明性が信頼になる。

## 2. データモデル(localStorage: `mintroom.weight.v1`)

```jsonc
{
  "startDate": "2026-07-01",     // ダイエット開始日(ユーザー設定)
  "baselineKg": 0,               // 開始日の体重(生値、ローカルのみ)
  "goalPercent": -8,             // 目標(相対%で保持。kg入力されてもアプリが即%変換し、目標kgも生値扱い)
  "entries": [{ "date": "2026-07-20", "kg": 0 }],  // 生値ログ(ローカルのみ)
  "shareMode": "relative"        // off | relative(既定) | ※絶対値共有モードは作らない
}
```

派生値(AIに渡してよい唯一の形。クライアント側で毎回計算):

```jsonc
{
  "percentFromBaseline": -2.5,   // 7日移動平均で計算、0.5%刻みに丸める
  "trend": "down",               // down | flat | up(直近7日 vs その前7日)
  "goalProgress": 31,            // 目標に対する進捗%(goalPercent 未設定なら null)
  "loggingDays": "6/7",          // 直近7日の記録日数(記録習慣の応援用)
  "daysSinceStart": 19
}
```

AIの文脈に注入される実際の文字列(サーバーの instructions 合成で1行):

```
[プライベート指標: 体重(相対値のみ共有)] 開始19日目 / 7日平均で -2.5% / 傾向: ゆるやか減少 / 目標進捗 31% / 記録 6/7日
```

**絶対に送らないもの:** kg、身長、BMI、体型・サイズ、目標kg。`shareMode: "off"` なら存在自体をAIに伝えない。

## 3. 丸めとノイズ対策(数値設計)

- **7日移動平均だけを使う。** 体重は1日で±1%平気で揺れる。日次の生の上下をAIにも本人にも強調しない(グラフは移動平均線を主、実測点は薄く)。
- **0.5%刻みに量子化して共有。** 細かすぎる数値は逆算の手がかりにもなるし、一喜一憂のもと。「-2.5%」で応援には十分。
- **±0.3%以内の変動は「flat」扱い。** 「増えた」という語をAIに使わせない閾値を仕様にする。

## 4. AIができること(相対値だけで十分成立する)

- 進捗の承認: 「開始から2.5%、着実だね」— 具体的だが数値羞恥がない。
- 停滞期の支え: trend が flat 継続 → 「停滞期はちゃんと減った人に来るやつだよ」(do_not_say: 原因追及・食事の詰問)。
- 記録習慣の応援: loggingDays ベースのソフト・ストリーク(habit メモ 2-1と同一機構)。
- 食事規則性との関連づけ: 食事ログ(habit メモ 2-3、これも「食べたかどうか」レベル)と合わせ、「朝ごはんが安定してた週は調子いいみたい」程度の緩い示唆まで。
- **やらないこと:** カロリー計算指導、目標体重の提案、他人との比較、医療・栄養アドバイス。

## 5. 安全設計(重要)

- **速すぎる減少は褒めない。** 7日平均で -1.5%/週を超えるペースが続いたら、賞賛ではなく「ちょっとペース速いかも。ちゃんと食べられてる?」という気づかいに切り替える(閾値は実装時に見直し)。減量アプリの最悪の失敗は不健康な減量の強化。
- 増加時に叱責・失望を出さない(do_not_say 固定)。flat/up が続いても「記録を続けてること自体」を承認対象にする。
- 摂食の乱れが疑われる会話が併発したら、体重の話題への言及自体を控える方向へ(orchestration の tone gate と同じ仕組みに載せる)。
- 医療機能ではないことを設定画面に明記(服薬チェックリストと同じ整理)。

## 6. UI(Life タブに1カード)

- **記録: 数字入力1つ+記録ボタンのみ**(1操作5秒原則)。前回値がプレースホルダ表示で打ち間違い検知。
- グラフ: 移動平均線メイン、y軸は実数kg(自分にだけ見える)。開始日と目標%のガイド線。
- 「AIビュー」トグル付きプレビュー: 「AIにはこう見えています: 開始19日目 / -2.5% / …」
- 共有設定はカード上に常時表示(埋もれさせない): 「AIと共有: しない / 相対値のみ」。既定は**相対値のみ**か**しない**か — 実装時にユーザーに確認(恥ずかしさ優先なら off 既定 + 初回記録時に1度だけ提案)。

## 7. 汎用化: プライバシー変換レイヤー

この「生値ローカル・派生値共有」パターンは体重専用にしない。共通インターフェイスにしておく:

```
PrivateMetric = {
  id, label,
  rawStore,                    // localStorage キー(生値)
  derive(raw) -> shared,       // AIに渡してよい派生値(相対%・傾向・進捗のみ)
  render(shared) -> string     // instructions 注入用の1行
}
```

将来同じ仕組みに載るもの: 体重以外のボディ指標、家計(金額を見せず「予算比◯%」だけ)、気分スコア(生スコアでなく傾向だけ)など。**「AIに相談したいが生数値は見せたくない」ものは全部この型。**

実装位置: 派生計算・丸めはクライアント(`public/` 側の小さな純関数モジュール)、注入文字列の合成はサーバーの instructions 合成(`server/openai.mjs`)— 生値はサーバーにすら送らない。純関数なのでユニットテスト対象(丸め・閾値・速すぎ判定は必ずテストを書く)。

## 8. 実装ステージ

| Stage | 内容 | AIへの共有 |
|---|---|---|
| W1 | Life タブに体重カード(記録・移動平均グラフ・開始日設定)。ローカル完結 | なし(shareMode: off のみ) |
| W2 | 派生値計算+AIビュー+shareMode: relative+instructions 注入 | 相対値のみ |
| W3 | ペース安全判定・停滞期サポート・食事ログ連携・週次ふりかえり統合 | 相対値のみ |
| W4 | PrivateMetric 汎用化(2つ目の指標を載せる時) | 指標ごとに opt-in |

W1 だけでも「記録グラフアプリ」として価値が立つ。W2 以降は既存のプライバシートグル群(Safety 設定)と同じ場所に統合すること。

## 9. ひとこと

「見られたくない(笑)」は仕様として正しい。恥ずかしさはダイエット継続の実際の障害で、**数値を見せない設計はふざけた要望ではなく、継続率を上げる合理的なプライバシー機能**。mint room の「優しい部屋」の核(責めない・失わない・嘘をつかない)に「**覗かない**」が加わる。
