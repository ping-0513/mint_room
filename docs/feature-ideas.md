# Feature ideas memo — 潜在ニーズメモ

Status: IDEAS ONLY(実装約束ではないブレインストーミングメモ)。実装時は `docs/agent-handoff.md` と `docs/4o-thinking-orchestration.md` の制約(ペイロード構築は `server/openai.mjs` に集約、DB/認証なし、モックは正直にラベル付け)に従うこと。

「こういうパーソナルアシスタントアプリを欲しがる人」が、口には出さなくても潜在的に必要としそうな機能のメモ。優先度の目安: ★=効果大かつ実装が軽い、☆=効果大だが重い、無印=あれば嬉しい。

## 1. チャット体験の質(いま一番効きやすい)

- ★ **Markdownレンダリング** — 現状は plain text 表示。箇条書き・コード・表が読めると体感が大きく変わる。目に優しい配色で。
- ★ **生成の停止ボタン(Stop)** — 長い返答を途中で止めたい。ストリーミング実装とセットで。
- ★ **メッセージのコピー ボタン** — レシピ、住所、手順などを他アプリへ持っていく導線。
- **自分のメッセージの編集 & 送り直し** — 打ち間違いのたびに新規送信するのはつらい。
- **複数の会話スレッド + 履歴一覧** — 「昨日の話どこいった?」問題。localStorage で `mintroom.chat.v1` を配列化するだけで始められる。
- **会話内検索 / 履歴検索** — 過去の会話が資産になる。
- **長い会話の自動要約で文脈維持** — `historyLimit` 超過分を要約して引き継ぐ(4o-thinking 設計の context_summary と共通化できる)。

## 2. 記憶とパーソナライズ

- ☆ **やさしい長期メモリ** — 「好きな食べ物」「呼ばれたい名前」「苦手なこと」をローカルに記憶し、会話に反映。何を記憶しているか一覧・削除できる透明性が必須(勝手に覚える不気味さを避ける)。
- **ペルソナプリセット** — 「はげまし系」「静かな執事」「元気な友達」などワンタップ切替。現状の persona 欄の上に乗せるだけ。
- ★ **日本語UIローカライズ** — ユーザーが日本語話者なら UI も日本語で。文言を一箇所に集めてから翻訳(i18n の巨大フレームワークは不要、辞書オブジェクトで十分)。

## 3. 生活管理の深掘り(Life タブ)

- ★ **タスクの期限・時刻** — ナッジ設計(routine_watch)の前提データにもなる。
- **繰り返しタスク / 毎日リセットされる服薬チェック** — 服薬は「毎朝リセット」が本来の姿。手動リセットは忘れる。
- **服薬の残量・補充リマインド** — 「あと3日分」で買い足しナッジ。医療アドバイスではなく在庫管理として。
- **買い物リストのカテゴリ分け・よく買うものテンプレ** — 定番品の再追加を1タップに。
- **ゆるい継続記録(streak)** — 罪悪感を煽らない設計で。「できなかった日」を責めない文言が大事。
- **起床・就寝の実績記録** — 目標時刻(既存)に対して「実際どうだったか」を軽く残せると振り返りができる。

## 4. 通知とナッジの実効性

- ☆ **ブラウザ通知 / PWA化** — タブを閉じていてもナッジが届かないと、就寝リマインドは実質機能しない。PWA + Notification API が MVP 的落とし所。プッシュサーバーはまだ不要。
- **おやすみモードの実体** — テーマ自動ディム、通知停止、翌朝のあいさつ予約。4o-thinking 設計の bedtime card の続き。
- **朝のあいさつカード** — 起床時間帯に「今日のタスク・予定・服薬」を1枚にまとめて表示。カレンダー嫌いなユーザーに一番効く可能性。

## 5. チャットから生活データを操作(tool use)

- ☆ **「牛乳買っておいて」→ 買い物リストに追加** — チャットが Life データを読み書きできると体験が一段変わる。OpenAI の function calling / tools を `server/openai.mjs` に追加し、実行はクライアントの localStorage 側で行う設計(サーバーに状態を持たない)。
- **「明日の予定は?」→ カレンダー参照して回答** — 同じ tool 基盤の読み取り側。
- 注意: 自動実行は追加系のみ、削除系は必ず確認を挟む。

## 6. 目と体への優しさ(アクセシビリティ)

- ★ **文字サイズ設定** — 目が疲れやすいユーザーには効果大。CSS 変数1個で済む。
- **アニメーション削減モード** — `prefers-reduced-motion` 対応 + 手動トグル(きらきら演出のOFF)。
- **輝度・コントラストの微調整** — ダークモードでも「真っ黒すぎる/明るすぎる」個人差がある。ミント系の中間トーンプリセット。
- **音声入力** — 疲れている時・寝る前にタイピングしたくない。Web Speech API ならローカルで試せる。
- **読み上げ(音声出力)** — 目を休めながら返答を聞ける。

## 7. データの安心感

- ★ **エクスポート / インポート(バックアップ)** — 全データが localStorage の現状、ブラウザのデータ消去で全消失する。JSON 1ファイルの手動バックアップが最小対策。DB より先にこれ。
- **「全データ削除」ボタン** — プライバシーの出口。何が消えるか明記。
- **API使用量・概算コストの表示** — 自分のAPIキーで動かす人は必ず気にする。レスポンスの usage フィールドを合算してローカル表示。
- **APIキーの接続テストボタン** — 設定画面から「キーが有効か」を1タップ確認(現状はサーバー再起動が必要な点も含め改善余地)。

## 8. 楽しさ・愛着(childish にしない範囲で)

- **部屋のコンパニオン演出** — 「mint room」の世界観として、時間帯で挨拶や背景の雰囲気が少し変わる程度の控えめな演出。キャラクター化はやりすぎない。
- **テーマバリエーション** — ミント基調は守りつつ、桜・ラベンダーなど差し色プリセット。
- **記念日・リマインド以外の「いいことメモ」** — アシスタントが「今日できたこと」を軽く承認してくれる欄。メンタルに優しいアプリの核になり得る。

## 実装順のおすすめ(効果 ÷ コスト順)

1. Markdownレンダリング + コピー + Stop(§1)
2. エクスポート/バックアップ(§7)— データ消失は信頼を一発で壊すため早めに
3. タスク期限 + 服薬の毎日リセット(§3)— ナッジ設計の前提データ
4. 文字サイズ + reduced motion(§6)
5. 日本語ローカライズ(§2)
6. PWA + ブラウザ通知(§4)— ナッジを実効化
7. tool use による Life 操作(§5)— 体験の飛躍だが設計レビュー必須

いずれも DB・認証・クラウド同期は不要のまま実装可能。§4 の PWA と §5 の tool use はスコープが太いので、着手前に `docs/agent-handoff.md` に段階計画を書いてから進めること。
